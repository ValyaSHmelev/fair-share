# FairShare — Техническое задание (ТЗ)

> Документ предназначен для **агентской разработки**: AI-агент должен реализовать приложение по этому ТЗ.

## 1. Обзор продукта

**FairShare** — веб-приложение для деления общего счёта между друзьями после совместных мероприятий (тусовки, выезды на природу).

**Проблема, которую решаем:** обычно платит один человек, а потом «раскидывает» сумму на всех. Но не каждую позицию нужно делить на всех (алкоголь — не на всех, мясо — не на всех и т.д.). Считать это вручную в Excel неудобно.

**Решение:** пользователь создаёт мероприятие, добавляет участников, заносит позиции расходов и для каждой позиции отмечает, **на кого её делить**. Приложение автоматически считает, сколько должен скинуть каждый участник, и показывает наглядный отчёт — как в Excel-табличках, но удобнее и редактируемо.

**Референс — реальные Excel-таблицы пользователя** (см. папку `examples/`):
- Столбцы: `Наименование`, `Цена`, далее по столбцу на каждого участника.
- В ячейке участника стоит его доля по позиции (если на него делили) либо пусто.
- Внизу строка `Итого` — сумма по каждому участнику.
- Деление всегда **поровну** между отмеченными участниками (например, мясо 1718 / 5 = 343,6).

## 2. Ключевые продуктовые решения (зафиксированы)

Эти решения приняты и не подлежат пересмотру агентом без явного указания:

1. **Модель учёта — «доли + плательщик»**. Для каждой позиции хранится, **кто её оплатил** (ровно один плательщик), и **на кого она делится**. Приложение считает: (а) сколько каждый участник должен скинуть в общий котёл (матрица долей, поведение Excel-таблиц) и (б) **оптимальную схему взаиморасчётов** «кто кому переводит» с минимальным числом транзакций (см. раздел 6.6–6.7).
   - Плательщик **обязателен** при создании/редактировании позиции; по умолчанию подставляется первый выбранный участник. Один плательщик на позицию (множественные плательщики — вне объёма).
   - Плательщик — это участник мероприятия; он не обязан входить в список тех, на кого делится позиция (например, оплатил, но за себя не считает).
2. **Деление позиции — только поровну** между выбранными участниками. Веса, проценты и точные суммы — НЕ реализуются.
3. **Участники и «кто за кого платит»**. Каждый участник — отдельная единица расчёта (доли считаются по-человечно).
   - **«Кто платит за участника» (`paidById`)**. У каждого участника можно указать, кто оплачивает его долю во взаиморасчётах. По умолчанию — **«Сам»** (`null`). Можно выбрать любого другого участника мероприятия (типичный кейс: парень платит за свою половинку). Это **не** меняет матрицу долей (раздел 6.5) — каждый по-прежнему считается отдельно, — но во **взаиморасчётах** (раздел 6.6–6.7) долю участника несёт его плательщик, что сокращает число транзакций. Ссылки разрешаются по цепочке до конечного плательщика; **циклы и висячие ссылки трактуются как «Сам»**.
4. **UI-библиотека — PrimeVue**. Тёмная тема и адаптивность обязательны.
5. **Округление — до целых рублей с компенсацией остатка** (метод наибольших остатков, см. раздел 6).
6. **Валюта — на уровне мероприятия**. При создании мероприятия выбирается валюта (по умолчанию ₽); внутри мероприятия валюта одна.
7. **Без бэкенда и без авторизации**. Все данные хранятся в `localStorage`. Поддерживается экспорт/импорт всех данных в JSON-файл.

## 5. Доменная модель данных

Все типы — в `src/types/models.ts`. Денежные суммы (`price`) хранятся как **число в основной денежной единице** (рубли/доллары), допускаются дробные значения при вводе; расчёты округляются по правилам раздела 6.

```ts
// Идентификаторы — строки UUID
export type ID = string

export type CurrencyCode = 'RUB' | 'USD' | 'EUR' | 'KZT' | 'GBP' // расширяемо

export interface Participant {
  id: ID
  name: string
  paidById: ID | null  // кто оплачивает долю участника во взаиморасчётах; null = «Сам»
}

export interface Expense {
  id: ID
  title: string                 // наименование позиции
  price: number                 // полная стоимость позиции
  payerId: ID | null            // кто оплатил (участник мероприятия); null допустим только для legacy-данных
  participantIds: ID[]          // на кого делится (поровну)
  createdAt: string             // ISO-дата
}

export interface FairEvent {
  id: ID
  name: string
  date: string | null          // ISO-дата мероприятия (опционально)
  currency: CurrencyCode
  participants: Participant[]
  expenses: Expense[]
  createdAt: string            // ISO
  updatedAt: string            // ISO
}

// Корневое состояние, сохраняемое в localStorage
export interface PersistedState {
  schemaVersion: number        // для будущих миграций; стартовое значение = 1
  events: FairEvent[]
}
```

**Инварианты модели:**
- `Expense.participantIds` ссылается только на существующих в этом мероприятии участников. При удалении участника его id удаляется из всех `participantIds` всех позиций.
- `Expense.payerId` ссылается на существующего участника мероприятия либо `null`. При удалении участника, который был плательщиком, `payerId` соответствующих позиций ставится в `null` (позиция выпадает из расчёта переводов до назначения нового плательщика).
- `Participant.paidById` ссылается на другого существующего участника мероприятия либо `null` («Сам»). Самоссылка (`paidById === id`) недопустима и нормализуется в `null`. При удалении участника, который был чьим-то плательщиком, у зависящих участников `paidById` ставится в `null`. Цепочки `paidById` разрешаются до конечного плательщика; циклы и ссылки на несуществующих участников трактуются как «Сам».
- Имена участников внутри одного мероприятия **могут повторяться** (тёзки — реальный кейс, см. «Алина» в `examples/тусовки - Новый год 2024.csv`). Идентификация — по `id`, не по имени.
- Позиция с пустым `participantIds` допускается при редактировании, но в отчёте подсвечивается как «ни на кого не разделено» (warning) и не участвует в итогах.
- Позиция с `payerId === null` допустима только для импортированных/legacy-данных: она попадает в матрицу долей, но **не участвует в расчёте взаиморасчётов** и подсвечивается предупреждением «не указан плательщик». В UI при создании/редактировании плательщик обязателен.

## 6. Алгоритмы расчёта (нормативно)

Все функции — чистые, без сайд-эффектов, в `src/lib/calc.ts`. Обязательно покрыть unit-тестами (Vitest).

### 6.1. Деление одной позиции поровну с округлением до рублей

Дано: `price` (число) и список из `n` участников (`n = participantIds.length`, `n > 0`).

Алгоритм (**метод наибольших остатков**, чтобы сумма долей точно равнялась `price`, округлённой до рубля):

1. Округлить `price` до целого рубля: `total = Math.round(price)`.
2. База на участника: `base = Math.floor(total / n)`.
3. Остаток: `rem = total - base * n` (целое, `0 <= rem < n`).
4. Каждому участнику назначается `base`. Затем `rem` участникам добавляется по `+1` руб.
   - Кому именно добавлять `+1`: первым `rem` участникам в **стабильном порядке** (порядок участников в массиве `participantIds`). Это делает результат детерминированным.
5. Итог: массив целочисленных долей, сумма которых строго равна `total`.

```ts
export function splitEqualRounded(price: number, n: number): number[] {
  const total = Math.round(price)
  if (n <= 0) return []
  const base = Math.floor(total / n)
  let rem = total - base * n
  const shares = new Array<number>(n).fill(base)
  for (let i = 0; i < n && rem > 0; i++, rem--) shares[i] += 1
  return shares
}
```

> Примечание: округляется **итоговая доля**, а распределение остатка гарантирует отсутствие расхождения «сумма долей ≠ стоимость позиции». Округление до рубля выбрано продуктово.

### 6.2. Итог по участнику за мероприятие

Сумма всех долей участника по всем позициям, где он отмечен:

```
totalForParticipant(p) = Σ по всем expense, где p ∈ expense.participantIds:
                           доля p в этой expense (из splitEqualRounded)
```

### 6.4. Итог по мероприятию

```
eventTotal = Σ Math.round(expense.price) по всем expense, у которых participantIds.length > 0
```

**Контрольный инвариант (must):** сумма всех `totalForParticipant` по всем участникам == `eventTotal`. Этот инвариант должен проверяться unit-тестом на сгенерированных и реальных (из `examples/`) данных.

### 6.5. Матрица отчёта

Отчёт строится как таблица:
- строки — позиции (`Expense`), столбцы — участники;
- в ячейке — доля участника по позиции (или пусто, если не участвует);
- нижняя строка `Итого` — `totalForParticipant` по каждому.

Это полностью воспроизводит формат Excel-таблиц пользователя.

### 6.6. Балансы участников (кто сколько заплатил и должен)

В расчёте взаиморасчётов учитываются только позиции, у которых одновременно `participantIds.length > 0` **и** `payerId !== null` (и `payerId` ссылается на существующего участника). Такие позиции называются **учтёнными**.

Для каждого участника `p` считаем по учтённым позициям. Долю каждого участника несёт его **конечный плательщик** `coverer(p)` — результат разрешения цепочки `paidById` (по умолчанию это сам участник, см. раздел 2, п.3):

```
paid(p)    = Σ Math.round(expense.price)   по учтённым позициям, где expense.payerId === p.id
owed(p)    = Σ доля q (из splitEqualRounded) по учтённым позициям и всем q ∈ expense.participantIds,
             у которых coverer(q) === p
balance(p) = paid(p) − owed(p)
```

> Если все участники платят «Сам» (`paidById === null`), формула вырождается в классическую: `owed(p)` равно сумме собственных долей `p`.

- `balance(p) > 0` — участнику **должны** вернуть (он переплатил, кредитор).
- `balance(p) < 0` — участник **должен** скинуть (дебитор).
- `balance(p) === 0` — в расчётах не участвует.

**Инвариант (must):** для каждой учтённой позиции сумма выданных долей строго равна `Math.round(price)` (раздел 6.1), а плательщику начисляется ровно `Math.round(price)`. Следовательно сумма всех `balance(p)` по всем участникам **строго равна 0**. Проверяется unit-тестом.

> Все суммы целочисленные (рубли), поэтому балансы и переводы — целые числа без накопления ошибок округления.

### 6.7. Оптимальная схема переводов (минимум транзакций)

Дано: набор балансов из 6.6. Нужно погасить все долги переводами «дебитор → кредитор» с **минимальным числом транзакций**.

Используется жадный алгоритм сведения долгов (стандартная практическая эвристика; точная минимизация — NP-трудная задача, но жадный подход даёт не более `k − 1` переводов, где `k` — число ненулевых балансов, и оптимален в большинстве реальных кейсов):

1. Разделить участников на **кредиторов** (`balance > 0`) и **дебиторов** (`balance < 0`, берём модуль).
2. Отсортировать обе группы **по убыванию суммы**; при равенстве — по `id` (детерминированность).
3. Пока есть и дебиторы, и кредиторы:
   - взять текущего дебитора `d` и кредитора `c`;
   - `amount = min(d.остаток, c.остаток)`;
   - если `amount > 0` — добавить перевод `{ fromId: d.id, toId: c.id, amount }`;
   - уменьшить остатки обоих на `amount`; того, у кого остаток стал `0`, перейти к следующему.
4. Результат — список переводов `Settlement[]`.

```ts
export interface Settlement {
  fromId: ID    // кто переводит (дебитор)
  toId: ID      // кому переводят (кредитор)
  amount: number // целое, > 0
}

export function minimizeTransfers(balances: Map<ID, number>): Settlement[] { /* см. реализацию */ }
```

**Инварианты (must, проверяются тестами):**
- Сумма всех `amount` == сумме положительных балансов == сумме модулей отрицательных балансов.
- После применения всех переводов итоговый баланс каждого участника равен 0.
- Число переводов ≤ (число ненулевых балансов − 1).
- Все `amount` целые и строго положительные.

### 6.8. Структура отчёта (расширение)

`buildReport(event)` дополнительно возвращает:
- `balances` — массив `{ participantId, paid, owed, balance }` в порядке участников;
- `settlements` — результат `minimizeTransfers`;
- `expensesWithoutPayer` — учтённые по участникам, но без плательщика позиции (предупреждение, не входят в переводы).

## 7. Хранилище, экспорт и импорт

### 7.1. localStorage
- Ключ: `fairshare:state:v1`.
- Стор Pinia (`events.ts`) при любой мутации сериализует `PersistedState` в `localStorage` (debounce ~200–300 мс или синхронно — на выбор, но без потери данных).
- При старте приложения состояние загружается из `localStorage`; если данных нет — пустой список мероприятий.
- **Миграция при загрузке:** для позиций без поля `payerId` (данные, созданные до появления плательщиков) значение нормализуется в `null`. Такие позиции остаются в матрице долей, но требуют назначения плательщика для участия во взаиморасчётах. Аналогично для участников без поля `paidById` (данные до появления «кто платит за участника») значение нормализуется в `null` («Сам»). Нормализация применяется и при импорте JSON, и при открытии shared-ссылок.
- Все операции записи обёрнуты в try/catch (обработка переполнения квоты / приватного режима) с уведомлением пользователя через PrimeVue `Toast`.

### 7.2. Экспорт
- Кнопка «Экспортировать данные» (в `SettingsView`) формирует JSON со всем `PersistedState` и скачивает файл вида `fairshare-backup-YYYY-MM-DD.json`.
- Формат файла:
```json
{
  "app": "FairShare",
  "schemaVersion": 1,
  "exportedAt": "2024-01-01T12:00:00.000Z",
  "data": { "schemaVersion": 1, "events": [ /* ... */ ] }
}
```

### 7.3. Импорт
- Кнопка «Импортировать данные» открывает выбор `.json`-файла.
- Валидация структуры (проверка наличия `data.events` и базовых полей). При несоответствии — `Toast` с ошибкой, импорт отменяется.
- Перед импортом — диалог подтверждения с выбором режима:
  - **Заменить** все текущие данные импортируемыми;
  - **Объединить** (добавить мероприятия из файла к текущим; конфликты `id` разрешать перегенерацией id у импортируемых).
- После успешного импорта — `Toast` об успехе и обновление списка.

## 8. Маршрутизация (Vue Router)

Многостраничный SPA, history mode.

| Путь | View | Назначение |
|---|---|---|
| `/` | `EventsListView` | Список мероприятий (главная) |
| `/events/new` | `EventCreateView` | Создание мероприятия (или диалог поверх `/`) |
| `/events/:id` | `EventDetailView` | Карточка мероприятия; редирект на `participants` |
| `/events/:id/participants` | `ParticipantsTab` | Участники |
| `/events/:id/expenses` | `ExpensesTab` | Позиции расходов |
| `/events/:id/report` | `ReportTab` | Итоговый отчёт |
| `/settings` | `SettingsView` | Экспорт/импорт, тема |
| `/:pathMatch(.*)*` | `NotFoundView` | 404 |

- Если мероприятие с `:id` не найдено — редирект на `/` с `Toast`.
- `EventDetailView` содержит заголовок мероприятия + навигацию по вкладкам (PrimeVue `TabMenu`/`Tabs`), под ней — `<router-view/>` для вложенных маршрутов.

## 9. Поэкранные требования (UI/UX)

Общие требования: тёмная тема по умолчанию + переключатель, адаптивность (мобильный → десктоп), компоненты PrimeVue, единый `Toast` для уведомлений, `ConfirmDialog` для удалений.

### 9.1. Шапка (`AppHeader`)
- Логотип/название «FairShare», ссылка на `/`.
- Переключатель темы (тёмная/светлая), значение хранится в `settings` сторе и `localStorage`.
- Ссылка на «Настройки» (`/settings`).

### 9.2. Список мероприятий (`EventsListView`)
- Карточки/список мероприятий: название, дата, число участников, общая сумма (`eventTotal`), валюта.
- Кнопка «Создать мероприятие».
- Действия на карточке: открыть, переименовать, дублировать `(опционально)`, удалить (с подтверждением).
- `EmptyState`, если мероприятий нет, с CTA «Создать первое мероприятие».
- Поиск/фильтр по названию `(опционально)`.

### 9.3. Создание мероприятия (`EventCreateView` / диалог)
- Поля: название (обязательно), дата (опционально, PrimeVue `DatePicker`), валюта (`Select`, по умолчанию ₽).
- После создания — переход на `/events/:id/participants`.

### 9.4. Участники (`ParticipantsTab`)
- Добавление участника: поле имени + кнопка «Добавить» (поддержать добавление по Enter; быстрый ввод нескольких).
- Список участников с возможностью переименовать и удалить.
- Удаление участника: подтверждение + предупреждение, что он будет убран из позиций; пересчёт отчёта.
- **Кто платит за участника (`paidById`):** при добавлении участника — `Select` «Кто платит» со значением по умолчанию «Сам» (можно выбрать любого из уже добавленных). У каждого участника в списке — `Select` для смены плательщика («Сам» + все, кроме него самого). Текущий выбор визуально отмечается тегом (напр. «платит Валя»). Это влияет только на секцию «Взаиморасчёты» отчёта (раздел 6.6–6.7), а не на матрицу долей.
- Тёзки допустимы — не блокировать одинаковые имена.

### 9.5. Позиции расходов (`ExpensesTab`)
Ядро приложения — здесь воспроизводится логика Excel-таблиц.
- Кнопка «Добавить позицию». Поля позиции: наименование, цена, **плательщик** (`Select` из участников, обязательно), выбор участников (`ParticipantPicker`).
- **Плательщик**: выпадающий список участников мероприятия. По умолчанию подставляется первый выбранный участник позиции. Сабмит блокируется, если плательщик не выбран. Если плательщик не входит в `participantIds` — это допустимо (заплатил, но за себя не делит).
- `ParticipantPicker`: список всех участников мероприятия с чекбоксами + кнопки «Выбрать всех» / «Снять всех». Показывать предварительную долю на человека (`price / n`, округлённую) в реальном времени.
- Таблица позиций: наименование, цена, **плательщик**, участники (теги/счётчик), доля на человека, действия (редактировать/удалить).
- Inline-редактирование `(опционально)`; как минимум — редактирование через диалог/раскрытие строки.
- Любое изменение позиции мгновенно отражается в отчёте (реактивность через стор).
- Предупреждение для позиций без участников.

### 9.6. Отчёт (`ReportTab`)
- Таблица в формате Excel: строки — позиции, столбцы — участники, ячейки — доли; нижняя строка `Итого` по каждому участнику.
- Блок суммарных итогов: общая сумма мероприятия (`eventTotal`), число позиций, число участников.
- **Секция «Взаиморасчёты» (см. 6.6–6.7):**
  - Блок балансов: по каждому участнику — заплатил (`paid`), доля (`owed`), баланс (`balance`) с визуальным выделением кредиторов (зелёный «+») и дебиторов (красный «−»).
  - Блок «Кто кому переводит» — список оптимальных переводов вида «`Имя А` → `Имя Б`: `сумма`». Если переводов нет (все в нуле) — сообщение «Все рассчитались».
  - Предупреждение, если есть позиции без плательщика (`expensesWithoutPayer`): они не учтены в переводах.
- Действия: **экспорт отчёта в CSV** (формат, совместимый с `examples/`), `печать (опционально)`.
- На мобильных таблица должна горизонтально скроллиться или перестраиваться в карточки по участникам.

### 9.7. Настройки (`SettingsView`)
- Экспорт всех данных (JSON), импорт (JSON) с режимами «Заменить»/«Объединить».
- Переключатель темы.
- Кнопка «Очистить все данные» (с двойным подтверждением).
- Версия схемы данных и краткая справка `(опционально)`.

- [ ] Экспорт данных в JSON и импорт (режимы «Заменить»/«Объединить») работают корректно.
- [ ] Экспорт отчёта в CSV формирует корректный файл.
- [ ] Тёмная тема включена по умолчанию, переключатель работает, верстка адаптивна от 360px.

## Приложение A. Проверка алгоритма на реальных данных

Контрольный пример из `examples/тусовки - шашлыки 290723.csv` (5 участников: Валя, Маша, Андр, Илья, Макс):

| Позиция | Цена | Делится на | Доля (округл. до руб.) |
|---|---:|---|---|
| сижки | 460 | Валя, Андр, Илья (3) | 154, 153, 153 |
| мясо | 1718 | все 5 | 344, 344, 343, 343, 344 (Σ=1718) |
| магнит | 771 | все 5 | 155, 154, 154, 154, 154 (Σ=771) |
| овощи | 169 | все 5 | 34, 34, 34, 34, 33 (Σ=169) |
| такси | 397 | Валя, Маша, Андр, Илья (4) | 100, 99, 99, 99 (Σ=397) |
| дастархан | 700 | все 5 | 140×5 (Σ=700) |

> Точное распределение «+1 руб» зависит от порядка участников в `participantIds` (см. 6.1), но **сумма по каждой позиции всегда точно равна округлённой цене**, а общий итог мероприятия сходится. Незначительные отличия в дробях от исходного Excel — следствие продуктового решения «округлять до рублей» (раздел 2, п.5).
