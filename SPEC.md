# FairShare — Описание приложения

## 1. Обзор продукта

**FairShare** — веб-приложение для деления общего счёта между друзьями после совместных мероприятий (тусовки, выезды на природу).

**Проблема:** обычно платит один человек, а потом «раскидывает» сумму на всех. Но не каждую позицию нужно делить на всех (алкоголь — не на всех, мясо — не на всех и т.д.). Считать это вручную неудобно.

**Решение:** пользователь создаёт мероприятие, добавляет участников, заносит позиции расходов и для каждой позиции отмечает, **на кого её делить**. Приложение автоматически считает, сколько должен скинуть каждый участник, показывает наглядный отчёт и формирует оптимальную схему переводов.

---

## 2. Ключевые продуктовые решения

1. **Модель учёта — «доли + плательщик»**. Для каждой позиции хранится, **кто её оплатил** (ровно один плательщик) и **на кого она делится**. Приложение считает: (а) сколько каждый участник должен скинуть (матрица долей) и (б) **оптимальную схему взаиморасчётов** «кто кому переводит» с минимальным числом транзакций.
   - Плательщик обязателен при создании/редактировании позиции; по умолчанию подставляется первый выбранный участник. Один плательщик на позицию.
   - Плательщик — участник мероприятия; он не обязан входить в список тех, на кого делится позиция.
2. **Деление позиции — только поровну** между выбранными участниками.
3. **«Кто платит за участника» (`paidById`)**. У каждого участника можно указать, кто оплачивает его долю во взаиморасчётах. По умолчанию — «Сам» (`null`). Это не меняет матрицу долей, но во взаиморасчётах долю участника несёт его плательщик — сокращает число транзакций. Ссылки разрешаются по цепочке до конечного плательщика; циклы и висячие ссылки трактуются как «Сам».
4. **UI-библиотека — PrimeVue**. Тёмная тема по умолчанию, адаптивность.
5. **Округление — до целых рублей** с компенсацией остатка методом наибольших остатков (сумма долей строго равна округлённой цене позиции).
6. **Валюта — на уровне мероприятия**. При создании выбирается валюта (по умолчанию ₽); внутри мероприятия валюта одна. Доступные: RUB, USD, EUR, KZT, GBP.
7. **Хранение и шаринг**. Данные хранятся в `localStorage`. Поддерживается экспорт/импорт всех данных в JSON-файл. Реализован шаринг отчёта по ссылке (данные мероприятия кодируются в URL через gzip + base64url).
8. **Авторизация — обязательная**. Вход через Firebase Authentication (email/password и Google). Доступ к приложению закрыт для неаутентифицированных пользователей; публичны только экраны входа/регистрации и просмотр отчёта по shared-ссылке. На текущем этапе данные мероприятий по-прежнему в `localStorage` (без привязки к пользователю); серверная синхронизация — будущая доработка.

## 5. Доменная модель данных

Все типы — в `src/types/models.ts`. Денежные суммы (`price`) хранятся как число в основной денежной единице; расчёты округляются до целых.

```ts
export type ID = string

export type CurrencyCode = 'RUB' | 'USD' | 'EUR' | 'KZT' | 'GBP'

export interface Participant {
  id: ID
  name: string
  paidById: ID | null  // кто оплачивает долю участника; null = «Сам»
}

export interface Expense {
  id: ID
  title: string
  price: number
  payerId: ID | null        // кто оплатил; null допустим только для legacy-данных
  participantIds: ID[]      // на кого делится (поровну)
  createdAt: string
}

export interface FairEvent {
  id: ID
  name: string
  date: string | null
  currency: CurrencyCode
  participants: Participant[]
  expenses: Expense[]
  createdAt: string
  updatedAt: string
}

export interface PersistedState {
  schemaVersion: number
  events: FairEvent[]
}

export interface ExportFile {
  app: 'FairShare'
  schemaVersion: number
  exportedAt: string
  data: PersistedState
}

export type ImportMode = 'replace' | 'merge'
```

**Инварианты модели:**
- `Expense.participantIds` ссылается только на существующих участников мероприятия. При удалении участника его id удаляется из всех `participantIds`.
- `Expense.payerId` при удалении участника-плательщика ставится в `null`.
- `Participant.paidById` ссылается на другого участника либо `null`. Самоссылка нормализуется в `null`. При удалении участника-плательщика у зависящих ставится `null`.
- Имена участников могут повторяться (тёзки). Идентификация — по `id`.
- Позиция с пустым `participantIds` не участвует в итогах и подсвечивается предупреждением.
- Позиция с `payerId === null` попадает в матрицу долей, но не участвует в расчёте взаиморасчётов и подсвечивается предупреждением.

## 6. Алгоритмы расчёта

Все функции — чистые, в `src/lib/calc.ts`, покрыты Vitest-тестами.

### 6.1. Деление позиции поровну с округлением (метод наибольших остатков)

Дано: `price` и `n` участников.

1. `total = Math.round(price)`
2. `base = Math.floor(total / n)`
3. `rem = total - base * n`
4. Каждому участнику назначается `base`; первым `rem` участникам (в порядке `participantIds`) добавляется `+1`.
5. Итог: массив целых долей, сумма строго равна `total`.

```ts
export function splitEqualRounded(price: number, n: number): number[] {
  const total = Math.round(price)
  if (n <= 0) return []
  const base = Math.floor(total / n)
  let rem = total - base * n
  const shares = new Array<number>(n).fill(base)
  for (let i = 0; i < n && rem > 0; i++, rem--) shares[i] += 1
  return shares
}
```

### 6.2. Итог по участнику

```
totalForParticipant(p) = Σ доля p по всем позициям, где p ∈ expense.participantIds
```

### 6.3. Итог по мероприятию

```
eventTotal = Σ Math.round(expense.price) по позициям с participantIds.length > 0
```

Инвариант: сумма всех `totalForParticipant` == `eventTotal`.

### 6.4. Матрица отчёта

Таблица: строки — позиции (`Expense`), столбцы — участники, ячейки — доля участника по позиции (или пусто), нижняя строка `Итого`.

### 6.5. Балансы участников

Учитываются только позиции с `participantIds.length > 0` и `payerId !== null`. Для каждого участника `p`:

```
coverer(p)  — конечный плательщик по цепочке paidById (по умолчанию сам p)
paid(p)     = Σ Math.round(expense.price) по позициям, где payerId === p.id
owed(p)     = Σ доля q по всем q, у которых coverer(q) === p
balance(p)  = paid(p) − owed(p)
```

- `balance > 0` — участнику должны вернуть (кредитор)
- `balance < 0` — участник должен скинуть (дебитор)
- Инвариант: сумма всех балансов строго равна 0.

### 6.6. Оптимальная схема переводов

Жадный алгоритм минимизации числа транзакций:

1. Разделить на кредиторов (`balance > 0`) и дебиторов (`balance < 0`), отсортировать по убыванию суммы.
2. Пока есть обе группы: взять текущего дебитора `d` и кредитора `c`, `amount = min(остаток_d, остаток_c)`, добавить перевод `{ fromId: d.id, toId: c.id, amount }`, уменьшить остатки.
3. Число переводов ≤ (число ненулевых балансов − 1).

```ts
export interface Settlement {
  fromId: ID
  toId: ID
  amount: number
}
```

## 7. Хранилище, экспорт и импорт

### 7.1. localStorage
- Ключ: `fairshare:state:v1`.
- Стор Pinia при любой мутации сохраняет `PersistedState` в `localStorage`.
- При загрузке: legacy-нормализация (`payerId` → `null`, `paidById` → `null` для старых записей).
- Операции записи обёрнуты в try/catch с уведомлением через PrimeVue `Toast`.

### 7.2. Экспорт JSON
Формирует и скачивает `fairshare-backup-YYYY-MM-DD.json`:
```json
{
  "app": "FairShare",
  "schemaVersion": 1,
  "exportedAt": "2024-01-01T12:00:00.000Z",
  "data": { "schemaVersion": 1, "events": [ /* ... */ ] }
}
```

### 7.3. Импорт JSON
- Валидация структуры; при ошибке — `Toast`.
- Два режима: **Заменить** все данные / **Объединить** (конфликты `id` разрешаются перегенерацией).

### 7.4. Шаринг отчёта по ссылке
- `share.ts` кодирует `FairEvent` в URL: JSON → gzip (CompressionStream) → base64url → `/share#{encoded}`.
- `SharedReportView.vue` декодирует данные из URL и отображает отчёт в режиме просмотра (без сохранения в localStorage).
- Нормализация legacy-полей применяется при декодировании.

| Путь | View | Доступ | Назначение |
|---|---|---|---|
| `/login` | `LoginView` | публичный | Вход |
| `/register` | `RegisterView` | публичный | Регистрация |

| `/events/:id/participants` | `ParticipantsTab` | защищённый | Участники |
| `/events/:id/expenses` | `ExpensesTab` | защищённый | Позиции расходов |
| `/events/:id/report` | `ReportTab` | защищённый | Отчёт |
| `/settings` | `SettingsView` | защищённый | Экспорт/импорт, тема |
| `/:pathMatch(.*)*` | `NotFoundView` | — | 404 |

### 8.1. Гард авторизации
- Авторизация **обязательна**. Публичные роуты (помечены `meta.public`): `/login`, `/register`, `/share`. Остальные — только для аутентифицированных пользователей.
- Глобальный `router.beforeEach`:
  - Дожидается первичной инициализации аутентификации (`onAuthStateChanged` разрешён хотя бы раз) через `authStore.ready`.
  - Если роут защищённый и пользователь не вошёл → редирект на `/login?redirect=<исходный_путь>`.
  - Если пользователь вошёл и открывает `/login` или `/register` → редирект на `/` (или на `redirect`).
- `/share` остаётся доступным без входа, чтобы можно было делиться отчётом.
- На данном этапе данные мероприятий продолжают храниться в `localStorage` и не привязаны к `uid` пользователя (привязка и синхронизация — будущие доработки).

## 9. Экраны (UI/UX)

Общее: тёмная тема по умолчанию + переключатель, адаптивность от 360px, компоненты PrimeVue, `Toast` для уведомлений, `ConfirmDialog` для удалений.

### 9.1. Шапка (`AppHeader`)
- Логотип «FairShare», ссылка на `/`.
- Переключатель темы (хранится в `settings` сторе).
- Ссылка на «Настройки».
- **Меню пользователя:** когда пользователь вошёл — показывается его email/имя (или `Avatar`) с выпадающим меню (`Menu`) и пунктом **«Выйти»** (`signOut` → редирект на `/login`).
- На публичных экранах входа/регистрации шапка не отображается.

### 9.2. Список мероприятий (`EventsListView`)
- Карточки: название, дата, число участников, общая сумма, валюта.
- Кнопка «Создать мероприятие» (диалог создания).
- Действия на карточке: открыть, переименовать, удалить (с подтверждением).
- `EmptyState` при отсутствии мероприятий.

### 9.3. Создание мероприятия
- Поля: название (обязательно), дата (опционально), валюта (по умолчанию ₽).
- После создания — переход на вкладку «Участники».

### 9.4. Участники (`ParticipantsTab`)
- Добавление участника: поле имени + кнопка «Добавить» (поддержка Enter).
- Список с переименованием и удалением (подтверждение + пересчёт отчёта).
- **«Кто платит за участника» (`paidById`):** `Select` «Сам» + все другие участники. Влияет только на взаиморасчёты, не на матрицу долей.
- Тёзки допустимы.

### 9.5. Расходы (`ExpensesTab`)
- Кнопка «Добавить позицию». Поля: наименование, цена, **плательщик** (обязательно), участники (`ParticipantPicker`).
- `ParticipantPicker`: список участников с чекбоксами, кнопки «Выбрать всех»/«Снять всех», предварительная доля на человека в реальном времени.
- Таблица позиций: наименование, цена, плательщик, участники, доля, действия (редактировать/удалить).
- Предупреждение для позиций без участников.

### 9.6. Отчёт (`ReportTab`)
- Таблица: строки — позиции, столбцы — участники, ячейки — доли; строка `Итого`.
- Блок суммарных итогов: общая сумма, число позиций, число участников.
- **Секция «Взаиморасчёты»:**
  - Балансы: заплатил / доля / сальдо с выделением кредиторов (зелёный «+») и дебиторов (красный «−»).
  - «Кто кому переводит» — список оптимальных переводов; если переводов нет — «Все рассчитались».
  - Предупреждение о позициях без плательщика.
- Экспорт отчёта в CSV, кнопка «Поделиться» (shared-ссылка).
- Таблица горизонтально скроллируется на мобильных.

### 9.7. Просмотр по ссылке (`SharedReportView`)
- Декодирует данные мероприятия из хэша URL.
- Показывает отчёт в режиме «только чтение».
- Кнопка «Сохранить к себе» для добавления мероприятия в localStorage.

### 9.8. Настройки (`SettingsView`)
- Экспорт данных (JSON), импорт (JSON) с режимами «Заменить»/«Объединить».
- Переключатель темы.
- Кнопка «Очистить все данные» (с двойным подтверждением).

### 9.9. Вход (`LoginView`)
- Центрированная карточка (`Card`) с логотипом «FairShare», без общей шапки приложения.
- Поля: **email** (`InputText`, тип email), **пароль** (`Password`, без индикатора силы).
- Кнопка **«Войти»** — `signInWithEmailAndPassword`. На время запроса — `loading`-состояние кнопки.
- Кнопка **«Войти через Google»** (иконка Google) — `signInWithPopup` с `GoogleAuthProvider`.
- Ссылка **«Нет аккаунта? Зарегистрироваться»** → `/register` (с сохранением `redirect`).
- Валидация: email непустой/корректный, пароль непустой. Ошибки Firebase (неверный email/пароль, пользователь не найден и т.д.) показываются через `Toast` с понятными русскоязычными сообщениями.
- После успешного входа — редирект на `redirect` или `/`.

### 9.10. Регистрация (`RegisterView`)
- Аналогичная карточка без общей шапки.
- Поля: **имя**, **фамилия** (в одной строке), **email**, **пароль** (`Password` с индикатором силы), **повтор пароля**.
- Кнопка **«Зарегистрироваться»** — `createUserWithEmailAndPassword`, затем `updateProfile` устанавливает `displayName = "Имя Фамилия"`, чтобы в интерфейсе (меню пользователя) отображалось имя, а не email (единообразно с входом через Google).
- Кнопка **«Регистрация через Google»** — `signInWithPopup` с `GoogleAuthProvider` (имя приходит из Google-профиля).
- Ссылка **«Уже есть аккаунт? Войти»** → `/login`.
- Валидация: имя и фамилия непустые, корректный email, пароль ≥ 6 символов (требование Firebase), совпадение паролей. Ошибки (`email-already-in-use` и др.) — через `Toast`.
- После успешной регистрации пользователь автоматически авторизован → редирект на `/`.

- `theme: 'dark' | 'light'` (по умолчанию `'dark'`), персист в localStorage.

### `stores/auth.ts`
- Состояние: `user: User | null` (текущий пользователь Firebase), `ready: boolean` (завершена ли первичная проверка сессии).
- Геттеры: `isAuthenticated`, `displayName` (`user.displayName || user.email`).
- Инициализация: подписка `onAuthStateChanged(auth, ...)` обновляет `user` и выставляет `ready = true` (вызывается один раз в `main.ts`).
- Действия (обёртки над `lib/auth.ts`): `loginWithEmail(email, password)`, `registerWithEmail(email, password, displayName?)`, `loginWithGoogle()`, `logout()`. При регистрации с указанным `displayName` вызывается `updateProfile`. Ошибки Firebase пробрасываются для обработки во вьюх. После `updateProfile` реактивность `displayName` принудительно обновляется (`triggerRef`).
- На данном этапе стор не влияет на `events` (данные общие в localStorage).
